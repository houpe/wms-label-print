# 批量编辑门店/收货人选择器改造设计

## 背景

当前首页已经将顶部页签文案展示为“单次新建”，单次新建中的温层数量也已经是左右 `- / +` 的步进器样式。现阶段的主要问题集中在批量上传后的编辑表格：

- “门店”和“收货人”使用 `react-select`，只能从已有数据中选择。
- 当 Excel 中的门店或收货人不在数据库里时，用户无法自然保留文本并继续编辑。
- 下拉面板依赖 `menuPortalTarget={document.body}` 和固定定位，存在层级和滚动容器适配问题。
- 用户明确要求不要继续手写选择器，也不要退回原生 `<datalist>`。

本次改造目标是用现成的第三方组件替换批量编辑表格中的两个选择器，并明确允许“数据库不存在的文本值”继续用于打印。

## 目标

- 将批量编辑表格中的“门店”和“收货人”改为现成的可搜索、可自由输入组件。
- 优先采用 Tailwind 体系内的方案，避免继续维护自定义弹层逻辑。
- 当输入命中已有数据时，自动回填关联字段。
- 当输入未命中已有数据时，保留输入文本，不阻止打印。
- 保持单次新建中的温层步进器交互不变。
- 统一文案，避免“单件新建 / 单次新建 / 新建”混用。

## 非目标

- 本次不新增“输入新门店/收货人后自动写入数据库”的能力。
- 本次不改造打印模板布局和打印逻辑。
- 本次不重构整页样式体系，不强行把现有页面整体迁移成纯 Tailwind 工具类写法。

## 方案比较

### 方案 A：Headless UI Combobox（推荐）

优点：

- 属于 Tailwind Labs 官方生态，和 Tailwind UI 的使用方式一致。
- 支持受控输入、模糊过滤、选项列表、自定义渲染。
- 允许“输入框值”和“选中项”分离，天然适合“命中则选中，未命中则保留文本”的场景。
- 组件结构清晰，可直接内嵌到表格单元格中，避免 `react-select` 的 portal 和 z-index 问题继续扩散。

缺点：

- 需要补一层轻量封装，统一处理输入文本、匹配项和回填行为。

### 方案 B：react-select Creatable

优点：

- 与现有依赖接近，替换成本较低。

缺点：

- 仍然依赖 `react-select` 的菜单定位与 portal 体系。
- 现有痛点中的弹层层级、滚动容器适配问题不会彻底消失。
- 表格内自由输入和匹配回填的行为更绕。

### 方案 C：原生 datalist

优点：

- 实现最简单。

缺点：

- 用户已明确不接受。
- 样式、键盘体验、跨浏览器表现都不稳定。

最终选型：`@headlessui/react` 的 `Combobox`。

## 交互设计

### 批量编辑表格

“门店”和“收货人”字段统一改为以下行为：

- 输入框默认展示当前单元格文本。
- 输入时按关键字过滤候选项。
- 选择已有项后，单元格值写入该选项名称。
- 没有命中候选项时，不报错，允许保留输入文本。
- 失焦后保留当前文本，不强制清空，也不强制回退到最近命中项。

### 门店字段

- 命中已有门店时：
  - `store` 更新为门店名。
  - 若该门店有地址，则用数据库地址覆盖当前行地址。
- 未命中已有门店时：
  - `store` 直接保留用户输入。
  - `address` 保持当前值不变，避免覆盖用户手动填写的地址。

### 收货人字段

- 命中已有收货人时：
  - `contact` 更新为收货人姓名。
  - 若该收货人有手机号，则用数据库手机号覆盖当前行电话。
- 未命中已有收货人时：
  - `contact` 直接保留用户输入。
  - `phone` 保持当前值不变。

### 单次新建

- 顶部页签继续使用“单次新建”文案。
- 温层区域继续使用左右 `- 数字 +` 步进器，不改成交互回退版本。
- 如果页面内还存在“单件新建”或容易引起误解的“新建”按钮文案，统一调整为更明确的文本：
  - 顶部页签：`单次新建`
  - 批量编辑入口按钮：`新增一行`

## 数据与状态设计

当前 `BatchRow` 结构可以继续保留：

```ts
interface BatchRow {
  id: number
  store: string
  contact: string
  phone: string
  date: string
  tempZone: string
  qty: number
  remark: string
  address: string
}
```

不新增数据库字段，也不修改打印接口。

需要补充的只是表格内组件状态：

- 每个组合框组件维护当前查询词 `query`。
- 组合框关闭时，将 `query` 同步回行数据字段。
- 选中已有项时，通过现有 `updateCell` 或新的辅助方法统一写回行数据。

建议将“门店回填逻辑”和“收货人回填逻辑”各自收敛成独立函数，避免散落在 JSX 回调中。

## 组件设计

新增一个表格内可复用组件，例如：

```tsx
type SearchableComboboxOption = {
  value: string
  label: string
  meta?: Record<string, string>
}
```

组件职责：

- 渲染输入框和候选面板。
- 根据输入词过滤候选项。
- 支持键盘上下选择与回车确认。
- 支持自由输入文本提交。
- 支持错误样式和占位文案。

组件不直接知道 `BatchRow` 结构，只通过以下接口与业务层交互：

- `value`
- `options`
- `placeholder`
- `hasError`
- `onValueCommit`
- `onOptionSelect`

这样可避免未来表格继续把业务逻辑和 UI 组件绑死。

## 校验规则

`validateRow()` 调整为业务必需字段校验，而不是“必须存在于数据库”校验：

- `store` 非空即可。
- `contact` 非空即可。
- `phone` 必填，且仍需满足手机号格式校验。
- `date` 必填。
- `tempZone` 必须为 `冷冻 / 冷藏 / 常温` 之一。
- `qty` 必须大于 0。

换句话说：

- “未命中数据库的门店/收货人”不再构成错误。
- 只有文本为空才算错误。

## 样式与层级设计

沿用现有 `globals.css` 的变量体系，不引入新的视觉风格。

组合框样式原则：

- 输入框高度、圆角、字号与现有 `.batch-input` 接近。
- 下拉面板使用白底、边框、阴影，与当前卡片风格一致。
- 面板定位优先相对当前单元格，不使用 `document.body` portal。
- 面板层级只需高于表格内容，不需要全局最高层，避免再次与模态框体系冲突。

建议新增样式：

- `.batch-combobox`
- `.batch-combobox__input`
- `.batch-combobox__panel`
- `.batch-combobox__option`
- `.batch-combobox__option--active`
- `.batch-combobox__option--selected`

- `src/app/page.tsx`

  - 新增可复用组合框组件或内联子组件。

  - 视情况移除未再使用的 `react-select` 依赖。

## 测试设计

按项目约定，本次功能改动必须补测试并在完成前执行测试。

建议增加以下自动化测试：

- 纯函数测试：
  - 门店命中已有数据时正确回填地址。
  - 门店未命中时保留文本且不覆盖地址。
  - 收货人命中已有数据时正确回填电话。
  - 收货人未命中时保留文本且不覆盖电话。
  - `validateRow()` 在“门店/收货人为自由输入文本”场景下返回通过。
- 组件交互测试：
  - 输入关键字可以过滤门店或收货人候选项。
  - 选择候选项后行数据被更新。
  - 输入不存在的值后失焦，文本保留。

如果当前仓库尚未配置测试运行器，本次实现需要一并补上最小可运行测试基础设施，优先选用适合 Next.js / React 19 的轻量方案，并确保测试命令可稳定执行。

## 验收标准

- 批量编辑里的“门店”和“收货人”都可搜索、可自由输入。
- 输入数据库中不存在的门店/收货人时，不再因为无法选择而卡住。
- 命中已有门店时能自动回填地址。
- 命中已有收货人时能自动回填电话。
- 行数据校验不再把“非数据库项”判为错误。
- 批量编辑不再依赖 `react-select` 菜单 portal 规避层级问题。
- 单次新建文案和温层步进器保持正确。
- 新增或更新的测试全部通过。

## 风险与回滚

主要风险：

- 表格内组合框的键盘交互需要在浏览器里验证，避免影响连续录入效率。
- 如果完全移除 `react-select`，要确认项目其他页面无依赖。

回滚策略：

- 如果新组合框在表格内交互不稳定，可临时保留 `react-select` 依赖但不再用于该表格。
- 功能改造将集中在单一页面文件和样式文件，回滚成本较低。
